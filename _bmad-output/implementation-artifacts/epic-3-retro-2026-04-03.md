The feedback list operates with audio, fading, and error persistence. Manual metadata editing with per-field confirmation on re-download is operational.

## Successes

- **Architecture scales cleanly** — MetadataProvider trait + ProviderRegistry + ChainExecutor allowed adding 7 providers without touching existing code (Open/Closed principle)
- **Infrastructure from Epic 1 holds** — service layer, soft-delete, optimistic locking, HTMX patterns scaled across 5 stories with zero structural refactoring

- **Stateless per-field confirmation** — MetadataResult serialized to hidden form fields avoids server-side session state
- **Cover cache busting** — `?v={timestamp}` pattern solves browser cache for re-downloaded covers

## Challenges

- **Batch shelving bug** — L-code scan with `last_volume_label` set shelved the last volume but did NOT activate batch mode, breaking the "catalog pile then shelve" workflow. Root cause: `if/else` branch returned early after shelving, never reaching batch mode activation. Fix: always activate batch mode on L-code scan, then shelve last volume if present.
- **HTMX fragment missing swap target** — `title_detail_fragment()` did not include `<div id="title-metadata">` wrapper, causing Edit/Redownload buttons to target a non-existent element. Fix: added wrapper div to fragment.
- **Cover URL update bypassed optimistic locking** — raw SQL UPDATE without version check discovered in code review pass 1, fixed with `rows_affected()` check.
- **MariaDB JSON/BLOB gotcha recurs** — `CAST(manually_edited_fields AS CHAR)` required again in story 3-5, exactly as in Epic 2.
- **Business logic leaked into route handlers** — story 3-5 initially had conflict detection, field comparison, and cache invalidation in `routes/titles.rs`. Refactored to `services/title.rs` during code review.

## Key Insights

1. **Test the real user flow, not isolated steps** — the batch shelving bug existed because tests only covered "L-code alone" or "V-code → L-code", never "catalog a pile → shelve the pile"
2. **HTMX fragments must mirror full-page template structure** — any `id` targeted by `hx-target` must exist in BOTH the full-page template AND the HTMX fragment
3. **Code review passes catch regressions from fixes** — each fix can introduce new bugs; iterative review is essential
4. **Synchronous re-download for user-facing confirmation** — unlike initial scan (fire-and-forget via tokio::spawn), re-download must block because the user needs to see the per-field confirmation dialog

## Bugs Found During Retrospective
