## Business Outcome

**Epic objective partially met.** Locations can be created, volumes shelved, and shelf contents browsed. However, the UX is not intuitive — Guy reported confusion about what to scan next and had to retry shelving operations.

## Successes

- Infrastructure from Epic 1 (service layer, soft-delete, locking) scaled smoothly — zero refactoring
- Tree rendering in Rust (not recursive Askama templates) solves compiler stack overflow — reusable pattern
- Code review multi-pass caught 9 patches across 3 stories (all Medium+ resolved before merge)
- Batch shelving mode works (L-code first → auto-shelve V-codes)
- Location breadcrumb with linked segments (get_path_segments)
- WITH RECURSIVE CTE for recursive volume counts
- MariaDB type gotchas documented in CLAUDE.md (BIGINT UNSIGNED, JSON/BLOB)

## Challenges

- **Shelving UX confusing** — scanning order (ISBN → V-code → L-code) not obvious, context lost between scans
- **No contextual guidance** — user doesn't know what to scan next
- **Titles not visible on home page** — FULLTEXT search may not find ISBN-only titles
- **MariaDB type mismatches** — BIGINT UNSIGNED and JSON/BLOB required CAST workarounds (3 occurrences)
- **Askama limitations** — recursive template inclusion crashes compiler, no `unwrap_or_default` filter
- **HX-Redirect ignored** on standard form POST (non-HTMX) — had to use HTTP redirect

## Key Insights

1. **Correct functionality ≠ good UX** — the app does the right thing but doesn't communicate it
2. **Contextual guidance is essential** — a message telling the user "what to do next" transforms usability
3. **MariaDB + SQLx type mapping** is a recurring pain point — document every pattern in CLAUDE.md
4. **Smoke tests must cover the real user journey** — not fragments with injected state
